## Project Overview

TypeDBClient_2.jl is a Julia FFI client for TypeDB 3.x, wrapping the official C shared library (`libtypedb_driver_clib`) via Julia's `ccall` interface. TypeDB 3.x removed sessions — transactions are opened directly on the driver with a database name.

## Commands

### Build (register the native library as an artifact)

```bash
# Option A – pre-built library
export TYPEDB_DRIVER_LIB=/path/to/libtypedb_driver_clib.dylib
julia -e 'using Pkg; Pkg.build("TypeDBClient")'

# Option B – build from source (requires Rust/Cargo)
export TYPEDB_DRIVER_SRC=/path/to/typedb-driver
julia -e 'using Pkg; Pkg.build("TypeDBClient")'
```

### Run Tests

```bash
# Unit tests (no server required)
julia --project -e 'using Pkg; Pkg.test("TypeDBClient")'

# Integration tests (requires TypeDB 3.x server)
TYPEDB_TEST_ADDRESS=localhost:1729 julia --project -e 'using Pkg; Pkg.test("TypeDBClient")'

# Run a single test file
julia --project test/unit/test_error.jl
```

## Architecture

### Layer Structure

```
User Code
    ↓
High-level API (driver.jl, database.jl, transaction.jl, query.jl, concept.jl)
    ↓ uses @checkerr macro + check_and_throw()
FFI sub-module (src/ffi/functions.jl)  ← all ccall wrappers
    ↓
libtypedb_driver_clib  (.dylib / .so / .dll)
```

### Key Design Patterns

**FFI Sub-Module (`src/ffi/functions.jl`)**: All `ccall` wrappers live in `module FFI` (a sub-module of `TypeDBClient`). It re-declares all handle type aliases locally (since Julia's grouped import syntax is invalid). Called as `FFI.driver_open(...)` etc. from all other source files.

**Handle Aliases**: All opaque C structs map to `Ptr{Cvoid}` with descriptive type aliases (e.g., `TypeDBDriverHandle`, `TransactionHandle`). These are declared in three places: `src/ffi/types.jl` (shared), `TypeDBClient.jl` (parent module scope), and `src/ffi/functions.jl` (FFI sub-module scope).

**Error Handling**: Every C API call that can fail must be followed by `check_and_throw()` or wrapped with `@checkerr`. The C driver uses a thread-local error state; `check_and_throw()` queries it and throws `TypeDBError` if set.

**String Ownership**:
- Borrowed strings (e.g., from `database_get_name`): use `typedb_string(cstr)` — no free needed
- Owned strings (e.g., from `database_schema`, `concept_get_string`): use `typedb_owned_string(cstr)` — calls `string_free` automatically

**Transaction Do-Block**: `transaction(driver, "dbname", TransactionType.WRITE) do tx ... end` auto-commits WRITE/SCHEMA on clean return, and rolls back + rethrows on exception. READ transactions are closed without committing.

**Artifact Resolution**: The library path is stored in `Artifacts.toml` as a content-addressed artifact. At module load (`__init__`), `_find_libtypedb()` reads the hash from `Artifacts.toml` and resolves the path. `deps/build.jl` registers the artifact.

### File Load Order (TypeDBClient.jl)

```
ffi/types.jl   → TransactionType, QueryType enums; handle aliases
ffi/functions.jl → module FFI with all ccall wrappers
error.jl       → TypeDBError, check_and_throw(), @checkerr
strings.jl     → typedb_string, typedb_owned_string
driver.jl      → TypeDBDriver struct + finalizer
database.jl    → Database struct + list/create/get/delete/schema ops
transaction.jl → Transaction struct + transaction() do-block
concept.jl     → Concept, ConceptRow, QueryAnswer, iterators
query.jl       → query()
exports.jl     → all public exports
```

### Query Answer Model (TypeDB 3.x)

`query(tx, typeql)` returns a `QueryAnswer` — check its type before consuming:
- Row queries (match/insert/select): `rows(answer)` → `ConceptRowIterator`
- Fetch/document queries: `documents(answer)` → `DocumentIterator` (JSON strings)
- Write/schema queries: `is_ok(answer)` → `Bool`

### Julia-Specific Gotchas

- TypeQL `$var` syntax in Julia strings must escape the `$`: `"match \$x isa thing;"`
- Triple-quoted docstrings also interpolate `$` — escape TypeQL variables there too
- Ternary operator needs a space before `?`: `Sys.iswindows() ? "a" : "b"` (not `iswindows()?`)
- `Project.toml` must declare stdlib deps: `Artifacts` + `Libdl` in `[deps]`; `Test` in `[extras]` + `[targets]`
- `runtests.jl` must `using TypeDBClient` before including sub-test files

### Yggdrasil/BinaryBuilder (build_tarballs.jl)

The `build_tarballs.jl` recipe builds `TypeDBDriverClib_jll`. Key facts:
- `name = "TypeDBDriverClib"` — BinaryBuilder appends `_jll` automatically
- `compilers = [:c, :rust]` required; `dont_dlopen = true` for Rust output
- `c/Cargo.toml` lacks `crate-type = ["cdylib"]` (auto-generated by Bazel) — fixed via `sed` in the build script
- `typedb-protocol` dependency requires `protoc` — handled via `HostBuildDependency("protoc_jll")`
- Cross-compilation target path variable: `${rust_target}` (not `${CARGO_BUILD_TARGET}`)
